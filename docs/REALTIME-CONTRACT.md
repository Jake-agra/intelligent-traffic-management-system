# Real-Time Contract

The backend is the authoritative source for both the web dashboard and mobile application.

## Initial WebSocket events

- `traffic.updated`
- `signal.updated`
- `violation.created`
- `incident.created`
- `incident.updated`
- `alert.created`
- `alert.acknowledged`
- `device.status_changed`
- `controller.mode_updated`

## Event envelope

Every domain event uses one consistent envelope:

```json
{
  "event": "traffic.updated",
  "version": 1,
  "event_id": "uuid",
  "occurred_at": "timezone-aware ISO timestamp",
  "intersection_id": "uuid or null",
  "data": {}
}
```

`event_id` is unique per event. `version` is the envelope schema version.
`occurred_at` is generated by the backend and must be timezone-aware.
`intersection_id` is `null` for system-wide events. `data` contains the
validated event payload for the event type.

## WebSocket endpoint

Clients subscribe at:

- `WS /api/v1/ws`

The backend sends a connection acknowledgement after accepting the socket. The
acknowledgement includes the connection identifier, supported events and the
applied subscription filters.

Optional filters:

- `intersection_id=<uuid>`
- `events=traffic.updated,signal.updated`

The backend rejects invalid subscription input before accepting the connection.

## Heartbeat

Clients may send:

```json
{ "type": "ping" }
```

The backend responds with:

```json
{ "type": "pong", "occurred_at": "timezone-aware ISO timestamp" }
```

This heartbeat is operational metadata and is not a domain event.

## Reconnection expectations

Clients should reconnect if the socket closes, then refresh current state through
the HTTP API before applying new live events. This prevents missed events during
disconnects from leaving web and mobile clients with stale state.

## Synchronization guarantees

- Web and mobile receive the same authoritative event payload.
- Commands are validated and persisted before broadcast where required.
- Clients reconnect and refresh current state after losing a connection.
- Event versions prevent stale updates from replacing newer state.
- Role permissions apply equally to web and mobile commands.

## MQTT-originated events

MQTT telemetry is normalized by the backend before any WebSocket publication.
Device heartbeats can emit `device.status_changed` when status changes. Traffic
telemetry emits `traffic.updated` after the reading is validated and stored.
Signal command acknowledgements emit `signal.updated` after the acknowledgement
payload is validated. MQTT handlers must use the shared realtime publisher rather
than broadcasting directly.

For physical signal synchronization, `signal.updated` distinguishes command
status from confirmed state:

- `status=requested` or `accepted`: command is pending; clients must not change
  authoritative signal colours yet.
- `status=executed` with `resulting_signals`: backend has persisted confirmed
  physical state and clients should refresh or apply the confirmed state.
- `status=rejected`, `failed` or `duplicate`: confirmed signal state is
  unchanged.

The event data may include `command_id`, `lane_id`, `requested_signal`,
`resulting_signals`, `device_id`, `source`, `confirmed`, `state_changed` and
`acknowledged_at`. Browsers consume this through the shared WebSocket provider;
they never subscribe to MQTT directly.

## Controller Mode Events

Phase 13.2 adds `controller.mode_updated` for confirmed or pending physical
controller mode and automatic phase state. Event data may include:

- `command_id`
- `status`
- `mode`
- `phase`
- `phase_started_at`
- `phase_duration_seconds`
- `next_phase`
- `device_id`
- `source`
- `confirmed`
- `state_changed`
- `acknowledged_at`

Clients treat `status=requested` or `accepted` as pending. They display manual
controls only after `mode=manual` is confirmed through the live API or a
confirmed realtime update. Automatic phase displays remain confirmed physical
state; the browser does not predict the next phase.

This contract prevents the web dashboard and mobile app from displaying conflicting traffic, signal, alert, incident or device information.
